Validation Guide

This guide captures a set of sanity checks to validate the synthetic omni-channel dataset generated by the retail datagen. Run the queries against the local DuckDB database at datagen-deprecated/data/retail.duckdb (for example with the duckdb CLI, opening the file read-only while the server is running).

There is also an API-level summary per table: GET /api/data/{table_name}/summary returns row counts and date ranges.

Checks

  • Pricing invariants
  • Cost < SalePrice <= MSRP for every product in dim_products

  • Cost / SalePrice between 0.50 and 0.85 for most products

  • Tax checks

  • Store tax_rate in a reasonable range (0–10.25%)
  • Reduced-rate items present in dim_products.taxability

  • Recompute sample receipt taxes: |recorded − computed| <= $0.02 for > 98% of receipts

  • POS behavior

  • Basket size by store_format roughly: express ~2 items, hypermarket 12–16

  • Returns present (receipt_type = 'RETURN') and negative quantities in fact_receipt_lines

  • Inventory fidelity

  • fact_store_inventory_txn includes SALE, INBOUND_SHIPMENT, RETURN, ADJUSTMENT reasons

  • fact_dc_inventory_txn includes INBOUND_SHIPMENT, OUTBOUND_SHIPMENT reasons

  • Online orders

  • Fulfillment mix near 60/30/10 (ship-from-DC / ship-from-store / BOPIS) in fact_online_order_headers
  • Lines carry promo_code 10–30% of the time

  • Small cancellation rate present; cancelled orders have zero totals

  • Payments

  • fact_payments rows reference either receipt_id_ext (in-store) or order_id_ext (online), never both

  • Small share of DECLINED payments with a decline_reason

  • Marketing

  • Device cost multipliers approximately 1.0x (mobile) / 1.2x (tablet) / 1.5x (desktop) across channels in fact_marketing

  • Presence / Sensors

  • Foot traffic zones non-uniform and vary by store_format
  • BLE pings per visit vary by format; roughly 30% of pings link to customers

Automated tests

The package's test suite covers most of these invariants:

cd datagen-deprecated
python -m pytest tests/
